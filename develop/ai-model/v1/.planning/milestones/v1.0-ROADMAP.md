# Roadmap: FastAPI AI Hub

## Overview

FastAPI AI 오케스트레이션 허브를 세 단계로 구축한다. Phase 1에서 서버 기반과 계약(스키마)을 확립하고, Phase 2에서 모든 외부 시스템 클라이언트(Pinecone, Embedding, Gemini, Router AI)를 안정화하고, Phase 3에서 실제 엔드포인트와 비동기 메모리 파이프라인을 완성한다. 각 단계는 독립적으로 검증 가능하며, 이전 단계가 안정되어야 다음 단계로 진행한다.

## Phase Details

### Phase 1: Foundation
**Goal**: WAS와의 API 계약이 코드로 확정되고, FastAPI 서버가 정상 기동한다
**Depends on**: Nothing (first phase)
**Requirements**: FOUND-01, FOUND-02, FOUND-03, FOUND-04
**Success Criteria** (what must be TRUE):
  1. `uvicorn` 실행 후 `GET /health` 요청이 200 OK를 반환한다
  2. `/process-meal`, `/recommend` 엔드포인트의 요청/응답 Pydantic 스키마가 존재하고, 잘못된 요청 본문에 422 에러를 반환한다
  3. `.env` 파일의 환경변수(API 키, Pinecone URL, WAS URL)가 누락되면 서버 시작 시 즉시 에러로 종료된다
  4. lifespan 이벤트에서 외부 클라이언트 초기화 코드가 실행 위치에 배치되어 있다 (실제 연결은 Phase 2에서 완성)
**Plans:** 2 plans
Plans:
- [x] 01-01-PLAN.md — 프로젝트 4계층 구조 + Settings + lifespan + /health
- [x] 01-02-PLAN.md — Pydantic 스키마 + 스텁 엔드포인트 (/process-meal, /recommend)

### Phase 2: Core Integrations
**Goal**: 모든 외부 시스템 클라이언트가 독립적으로 동작하고 테스트 가능하다
**Depends on**: Phase 1
**Requirements**: PINE-01, PINE-02, PINE-03, PINE-04, EMBD-01, EMBD-02, EMBD-03, GEMI-01, GEMI-02, GEMI-03, GEMI-04, GEMI-05, ROUT-01, ROUT-02, ROUT-03
**Success Criteria** (what must be TRUE):
  1. PineconeAsyncio 클라이언트가 user_id namespace로 벡터를 저장하고 검색한다. 두 개의 다른 user_id로 저장한 벡터가 서로 검색되지 않는다
  2. 한국어 텍스트를 입력하면 384차원 임베딩 벡터가 반환된다. 임베딩 생성이 async 함수 내에서 event loop를 블록하지 않는다
  3. Gemini Flash 클라이언트가 Mode 7(식단 분석)과 Mode 8(추천) 프롬프트로 호출 시 지정된 JSON 스키마 형식의 응답을 반환한다
  4. Router AI가 테스트 입력 문장에 대해 6가지 모드 중 하나와 reason을 JSON으로 반환한다
  5. Gemini API가 429 에러를 반환할 때 exponential backoff로 재시도한다
**Plans:** 5/5 plans complete
Plans:
- [x] 02-01-PLAN.md — Settings 환경변수 + 의존성 + EmbeddingClient
- [x] 02-02-PLAN.md — PineconeClient (async 검색/저장, namespace 격리)
- [x] 02-03-PLAN.md — GeminiClient + Mode 7/8 프롬프트
- [x] 02-04-PLAN.md — RouterClient + 시스템 프롬프트
- [x] 02-05-PLAN.md — Lifespan 초기화/종료 완성

### Phase 3: Endpoints and Memory
**Goal**: WAS가 /process-meal과 /recommend를 호출하면 AI 분석 결과를 받고, 응답 후 대화 맥락이 Pinecone에 비동기로 저장된다
**Depends on**: Phase 2
**Requirements**: MEAL-01, MEAL-02, MEAL-03, MEAL-04, MEAL-05, RECOM-01, RECOM-02, RECOM-03, RECOM-04, RECOM-05, BGSM-01, BGSM-02, BGSM-03, BGSM-04, BGSM-05
**Success Criteria** (what must be TRUE):
  1. `POST /process-meal`에 user_id, user_profile, user_message를 포함한 요청을 보내면 `{status, data: {calories, message}}` 형식의 응답이 반환된다
  2. `POST /recommend`에 user_id, user_profile, user_instruction을 포함한 요청을 보내면 `{status, data: {recommended_exercise: {name, burn_calories}, recommended_meal: {name, calories}}}` 형식의 응답이 반환된다
  3. 두 엔드포인트 모두 응답을 반환한 뒤 Background Summary가 비동기로 실행되어 Pinecone의 해당 user_id namespace에 벡터가 upsert된다
  4. Background Summary 파이프라인에서 에러 발생 시 에러 로그가 기록되고 메인 응답에는 영향이 없다
  5. 동일 user_id로 여러 번 요청 후 Pinecone 검색 시 이전 대화 맥락이 응답에 반영된다
**Plans:** 3/3 plans complete
Plans:
- [x] 03-01-PLAN.md — Summary 프롬프트 + Background Summary 파이프라인 + 프롬프트 빌더 확장
- [x] 03-02-PLAN.md — /process-meal 서비스 파이프라인 + 글로벌 예외 핸들러
- [x] 03-03-PLAN.md — /recommend 서비스 파이프라인
