### fast_api 

## 라우터 ai 
# input : 라우터 시스템 지침서 + 사용자 메세지
# output : 선택된 모드 + 선택된 근거
++ 식단기록과 추천 기능은 라우터 사용하지 않음 
++ 식단기록은 식단 기록 버튼 입력 시 writing_meal_flag:1 모드로 전환하기
++ 추천기능은 백그라운드 실행 및 새로 고침 시 (추천 수용 버튼 입력 시 단순 플래너에만 적용되고 ai 사용 안함)

## 워커 ai 
# input : 선택된 모드마다 다름 
총 구조 : 사용자 메세지 + 사용자 지시사항 + 시스템 지시사항 + 사용자 정보 + 이전 대화 기록 (벡터 db) + 라우터 결과값 + 기타
0. 공통 값 : 
사용자 메세지 (1순위)
사용자 지시사항 (2순위)
시스템 지시사항 (3순위) : 아웃풋 값의 json 형식 지정 및 모드 별 특이사항 전달
사용자 정보 (db) + 이전 대화 기록 (벡터_db) 

1. 단순대화 : 공통값 + 라우터 ai 결과 값 
2. 플랜 작성 : 공통값 + 라우터 ai 결과 값 
3. 플랜 수정 : 공통값 + 라우터 ai 결과 값 + 현재 운동 항목 리스트를 fast_api가 백엔드에요청 
4. 식단 작성 : 공통값 + 라우터 ai 결과 값 
5. 식단 수정 : 공통값 + 라우터 ai 결과 값 + 현재 식단 리스트를 fast_api가 벡엔드에 요청
7. 식단 기록 : 공통값
8. 추천 기능 : 공통값


# output (시스템 지시사항과 사용자 지시사항, 사용자 메세지에 맞게 반환되는 값) 
db_modified_flag는 단순 대화 제외 모두 1로 설정 (** 논의 사항 : 그냥 fast_api 알고리즘 상으로 특정 모드 처리하도록 하는게 직관적으로 보임)
1. 단순대화 : 질문답변
2. 플랜 작성 : { 날짜, 운동 종류, 운동 세부 항목, 횟수 } + 디폴트 값은 일주일 
3. 플랜 수정 :  { 날짜, 운동 종류, 운동 세부 항목, 횟수 } + 디폴트 값은 전체 수정 
4. 식단 작성 : 날짜 + 식품 종류 + 시간 (디폴트 아 점 저)
5. 식단 수정 : 날짜 + 식품 종류 + 시간 (디폴트 전체 수정 디폴트)
7. 식단 기록 : 칼로리 + 메세지 (한 줄로 최대한 간결히)
8. 추천 기능 : 추천할 운동, 운동 소모 칼로리, 추천할 식단, 식단 칼로리



** 추가 논의 사항 : 
1. 라우터 ai를 메인 ai로 변경 후 시간 오래 걸리는 작업이라고 판단하면 운동/식단 전문가 ai(현 워커 ai)로 옮기기 
아니면 병렬로 일반 ai(단순 대화, 사용자 db 수정, 식단기록)와 전문가 ai(플랜/식단/추천기능)로 나누기
(이유 : 라우터 ai를 좀 더 사용하는 방향)
