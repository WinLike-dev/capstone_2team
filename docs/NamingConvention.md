# 📄 프로젝트 문서 네이밍 규칙

이 프로젝트의 기술 문서는 **두 가지 계열**로 구성됩니다.

| 계열 | 접두사 | 용도 |
| :--- | :--- | :--- |
| **DataFormat** | `DataFormat_` | 데이터 규격 명세 (JSON 포맷, API 엔드포인트, DB 스키마, 상태 구조) |
| **Sequence** | `Sequence_` | 시퀀스 다이어그램 (Mermaid 기반 흐름도) |

---

## 1. 공통 규칙

### 1.1 파일명 구조

```
{접두사}_{탭번호|역할}[-{시퀀스}]_{레이어}.md
```

| 구성 요소 | 필수 | 설명 | 형식 |
| :--- | :--- | :--- | :--- |
| `{접두사}` | ✅ | 문서 계열 (`DataFormat` 또는 `Sequence`) | 고정 문자열 |
| `{탭번호\|역할}` | ✅ | 앱 탭 인덱스(0~3) 또는 특수 역할명 | 숫자 또는 `index`, `overview` |
| `{시퀀스}` | ❌ | 같은 탭·같은 레이어에서 문서가 2개 이상일 때 부여 | `-숫자` (예: `-2`, `-3`) |
| `{레이어}` | ✅ | 문서가 다루는 시스템 레이어 | `_api`, `_ai`, `_front`, `_db` |

> [!NOTE]
> **특수 역할명**
> - `index` — 해당 계열의 **목차·인덱스** 문서 (하위 문서 링크 모음)
> - `overview` — **특정 탭에 귀속되지 않는 전역 흐름** 문서

> [!NOTE]
> **시퀀스 번호 규칙**
> - 같은 탭 + 같은 레이어의 첫 번째 문서에는 시퀀스를 붙이지 않습니다.
> - 두 번째부터 `-2`, `-3` 순으로 붙입니다.
> - 예: `DataFormat_1_api.md` → `DataFormat_1-2_api.md` → `DataFormat_1-3_api.md`

### 1.2 레이어 접미사 정의

분류 기준: **"데이터가 어느 구간을 흐르는가"**

```
Front ←→ WAS(Backend) ←→ AI(FastAPI)
           ↕
          DB
```

| 접미사 | 다루는 구간 | 포함 내용 | 대상 독자 |
| :--- | :--- | :--- | :--- |
| **`_api`** | **Front ↔ WAS** | 엔드포인트, 요청/응답 JSON, DB 스키마, 비즈니스 시퀀스 | 프론트 · 백엔드 |
| **`_ai`** | **WAS ↔ AI(FastAPI)** | AI 요청/응답 JSON, 라우팅 로직, 모드 분류, 벡터DB | AI · 백엔드 |
| **`_front`** | **Front 내부** | 전역 상태(Context), DTO, UI 연동 흐름, 렌더링 로직 | 프론트 |
| **`_db`** | **DB 단독** | 테이블 설계, ERD, 마이그레이션 (API 없이 DB만 다룰 때) | 백엔드 · DBA |

> [!TIP]
> **`_api` vs `_ai` 판단법**
> - AI(FastAPI)가 **개입하지 않는** 흐름 → `_api` (Front → WAS → DB)
> - AI(FastAPI)가 **개입하는** 흐름 → `_ai` (WAS ↔ AI, 또는 Front → WAS → AI → WAS → Front)

### 1.3 탭 번호 매핑

| 탭 번호 | 탭 이름 | 주요 기능 |
| :--- | :--- | :--- |
| `0` | 온보딩 / 프로필 | 사용자 건강 정보·성향 초기 설정 |
| `1` | 홈 (대시보드) | AI 추천 표시, 추천 항목 추가·교체, 식단 기록 |
| `2` | 채팅 | AI 채팅 기반 플랜 수정·상담 |
| `3` | 캘린더 | 일정 확인, 완료 체크, 달성률 추적 |

## 3. Sequence 계열

시스템 간 **흐름과 상호작용**을 Mermaid 시퀀스 다이어그램으로 시각화하는 문서입니다.

### 특수 파일

| 파일명 | 역할 | 설명 |
| :--- | :--- | :--- |
| `Sequence_index.md` | 목차 | 전체 시퀀스 문서의 인덱스 + 전역 개요 다이어그램 |
| `Sequence_overview_ai.md` | 전역 흐름 | 특정 탭에 귀속되지 않는 전체 AI 통신 흐름 |

### 탭별 파일

| 파일명 | 탭 | 레이어 | 내용 요약 |
| :--- | :--- | :--- | :--- |
| `Sequence_0_api.md` | 탭0 (프로필) | API | MBTI 데이터 저장 흐름 (AI 미사용) |
| `Sequence_1_ai.md` | 탭1 (홈) | AI | 식단 기록·추천 AI 프로세스 상세 흐름 |
| `Sequence_2_ai.md` | 탭2 (채팅) | AI | AI 채팅 라우팅·의도 분류 상세 흐름 |

## 4. 새 문서 작성 가이드

### Step 1: 계열 결정
→ 데이터 규격(JSON, 스키마)인가, 흐름도(시퀀스 다이어그램)인가?

| 문서 성격 | 접두사 |
| :--- | :--- |
| JSON/API/상태 구조 정의 | `DataFormat_` |
| Mermaid 시퀀스 다이어그램 | `Sequence_` |

### Step 2: 탭 번호 또는 역할 결정
→ 특정 탭에 해당하는가, 아니면 전역/목차인가?

| 해당 범위 | 값 |
| :--- | :--- |
| 특정 탭 기능 | `0`, `1`, `2`, `3` |
| 전역 흐름 (탭 무관) | `overview` |
| 목차·인덱스 | `index` |

### Step 3: 레이어 결정
→ 데이터 흐름의 주 구간이 어디인가?

| 주 구간 | 접미사 |
| :--- | :--- |
| Front ↔ WAS 간 통신 | `_api` |
| WAS ↔ AI(FastAPI) 간 통신 | `_ai` |
| Front 내부 상태·UI | `_front` |
| DB 스키마만 단독 정의 | `_db` |

### Step 4: 시퀀스 번호 확인
→ 같은 탭 + 같은 레이어의 문서가 이미 존재하는가?

- 없으면 → 시퀀스 없이 작성 (예: `DataFormat_2_api.md`)
- 있으면 → 다음 시퀀스 부여 (예: `DataFormat_2-2_api.md`)

### 네이밍 예시

```
DataFormat_2_api.md           → 탭2(채팅) Front↔WAS API 규격
DataFormat_2-2_api.md         → 탭2(채팅) Front↔WAS API 규격 (두 번째)
DataFormat_3_api.md           → 탭3(캘린더) API 규격
Sequence_3_api.md             → 탭3(캘린더) 비즈니스 흐름도
Sequence_3_ai.md              → 탭3(캘린더) AI 연동 흐름도
Sequence_overview_front.md    → 전역 프론트 흐름도
```

> [!IMPORTANT]
> **네이밍 체크리스트**
> 1. 접두사가 `DataFormat_` 또는 `Sequence_`로 시작하는가?
> 2. 탭 번호(0~3) 또는 역할명(`index`, `overview`)이 올바른가?
> 3. 레이어 접미사(`_api`, `_ai`, `_front`, `_db`)가 문서 내용과 일치하는가?
> 4. 같은 탭·레이어에 기존 문서가 있다면 시퀀스 번호를 붙였는가?
